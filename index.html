§5 Comparison Report

Before Defence vs. After Defence — Technical analysis of DNS-over-HTTPS efficacy against local-area network DNS spoofing attacks.

Executive Summary

This report analyzes the efficacy of DNS-over-HTTPS (DoH) in mitigating local-area network (LAN) DNS spoofing attacks. In the "Before" scenario, the network relied on traditional plaintext DNS (UDP Port 53). In the "After" scenario, cryptographic encapsulation via HTTPS (TCP Port 443) was implemented at the application layer.

Technical Analysis: Before Defence

During the initial phase, the Attacker utilised the dnsspoof tool from the dsniff package to monitor the local network segment for DNS queries. Because standard DNS queries are transmitted as unencrypted UDP packets and lack any form of authentication or integrity verification, the attacker was able to perform a classic "race condition" attack.

When the Victim initiated a DNS query for test.local, the request was broadcast toward the upstream DNS resolver (8.8.8.8). The Attacker, running in promiscuous mode, observed this query in real-time and injected a forged response with IP 192.168.1.50 before the legitimate reply arrived. The Victim accepted the first response, cached the malicious mapping, and loaded the attacker's webpage.
BEFORE DEFENCE — DNS Flow (Plaintext UDP:53)
victim@win:~$ ping test.local DNS query sent → 192.168.1.1 (UDP port 53) Attacker intercepts query in promiscuous mode dnsspoof injects forged response: test.local → 192.168.1.50 Victim caches malicious DNS response Pinging test.local [192.168.1.50] ← ATTACKER'S IP! HTTP 200 OK from malicious server

Technical Analysis: After Defence

Upon enabling DNS-over-HTTPS in the browser, the Victim fundamentally altered its DNS resolution behaviour. Instead of broadcasting plaintext UDP queries to the local network's default gateway, the browser established an encrypted TLS 1.3 tunnel directly to a trusted remote resolver (Cloudflare's 1.1.1.1 service).

The query was encapsulated within an HTTPS POST request to https://cloudflare-dns.com/dns-query, traveling over TCP port 443, encrypted with the same cryptographic protections used for secure web browsing. The dnsspoof tool reported zero pattern matches — no plaintext DNS traffic traversed the network.
AFTER DEFENCE — DNS Flow (HTTPS/TLS 1.3 port 443)
victim@win:~$ ping test.local DoH query → HTTPS POST to cloudflare-dns.com:443 TLS 1.3 encrypted tunnel established Attacker sees: encrypted HTTPS traffic only dnsspoof: 0 pattern matches (no UDP port 53 queries) Cloudflare returns: NXDOMAIN (test.local doesn't exist) DNS Spoofing: BLOCKED ✓

Why the Defence Was Effective — Three Technical Factors

🔐
Encryption
The attacker could no longer observe DNS request contents. Domain names are encrypted within the TLS payload. Network capture reveals only encrypted HTTPS traffic.
🛡️
Integrity Protection
TLS provides cryptographic message authentication through HMAC verification. Injected malicious data would be detected via MAC failures, terminating the connection.
🌐
Trust Boundary Relocation
DoH bypasses the local router's DNS configuration, moving the trust boundary from the untrusted LAN to a known, authenticated secure endpoint.

Conclusion

The implementation of DNS-over-HTTPS represents an effective application-layer defence against local network DNS spoofing attacks. By encrypting queries and authenticating responses through TLS, DoH eliminates the attacker's ability to observe, intercept, or modify DNS traffic. Organizations and individual users should consider enabling DoH as a baseline security measure, particularly when operating on untrusted network segments such as public Wi-Fi or shared corporate networks.
